# RLS Depth Audit — 2026-05-18

Static-code audit of Row Level Security policies on sensitive tables, scoped to `docs/plans/2026-05-18-production-readiness-stabilization.md` Task 10. Reviews policy text and identifies gaps; does NOT validate against the production database (that requires psql access).

## Scope

Sensitive tables explicitly enumerated in the readiness plan:
- `ledger_entries` (finance source of truth)
- `chef_payout_accounts` (Stripe Connect bridge)
- `driver_documents` (PII — driver's licence, insurance)
- `order_exceptions` (operational signals)
- `support_threads`, `support_messages` (customer/staff dialogue)
- `audit_log` (governance trail)

## Method

`grep "CREATE POLICY.*<table>" supabase/migrations/*.sql` for each table; cross-read with `docs/AUTH_ROLE_MATRIX.md` for the expected role-set.

## Findings

### ✅ `ledger_entries` — adequate

| Migration | Policy | Roles | Operation |
|-----------|--------|-------|-----------|
| `00007_central_engine_tables.sql:419` | `ledger_entries_finance_view` | `ops_agent, ops_manager, finance_admin, super_admin` | SELECT |
| `00007_central_engine_tables.sql:429` | `ledger_entries_insert_system` | authenticated (any) | INSERT (system service-role only via API guard) |
| `00025_rls_role_alignment.sql:171` | Re-grants `ledger_entries_finance_view` via `is_finance_staff(auth.uid())` helper | finance + ops roles | SELECT |

The INSERT-`WITH CHECK (true)` policy looks loose but is acceptable because all writes flow through the engine's `LedgerService` via service-role admin clients (which bypass RLS); customer/chef/driver sessions cannot reach the INSERT path through any documented route. A future hardening would be to scope INSERT to a specific service-role check.

### ✅ `chef_payout_accounts` — adequate

### ✅ `order_exceptions` — adequate

| Migration | Policy | Roles | Operation |

### ⚠️ `driver_documents` — GAP: no ops-read policy

| Migration | Policy | Roles | Operation |
|-----------|--------|-------|-----------|
| `00002_rls_policies.sql:272` | `Drivers can manage own documents` | driver-owner | ALL |
| **(missing)** | — | — | — |

**No ops-read policy exists.** Ops cannot view `driver_documents` rows via PostgREST. The chef analog (`chef_documents` if it exists) would need the same check.

**Impact:** Driver document review (Gate 5 of `PILOT_CHEF_LAUNCH_PLAN_2026-05-14.md`) requires ops to inspect licence + insurance documents. With no SELECT policy for platform staff, the ops-admin UI must either:
- Use a service-role admin client to bypass RLS (current pattern in `apps/ops-admin/src/lib/engine.ts` likely does this — verify)
- OR a policy must be added.

If the admin client is used, this is technically secure (service role is privileged) but breaks the principle of least privilege and obscures the access pattern. Best practice: add an explicit `Ops can read driver documents` policy.

**Recommended migration** (to be authored as `supabase/migrations/00039_driver_documents_ops_read.sql` once Sean approves):

```sql
DROP POLICY IF EXISTS "Ops can read driver documents" ON driver_documents;
CREATE POLICY "Ops can read driver documents" ON driver_documents
  FOR SELECT TO authenticated
  USING (
    public.is_platform_staff(auth.uid())
    OR EXISTS (
      SELECT 1 FROM drivers
      WHERE drivers.id = driver_documents.driver_id
      AND drivers.user_id = auth.uid()
    )
  );
```
