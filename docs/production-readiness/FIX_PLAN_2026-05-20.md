## P0 - Before Peer Review

### 1. Partial refund payment status mismatch

Issue: Partial refund path writes `orders.payment_status = 'partial_refunded'`, but the database check constraint does not allow that value.
Impact: Partial refund webhook or ops refund flow can fail after Stripe succeeds, leaving order, payment, ledger, and refund state inconsistent.
Files involved:
- `packages/engine/src/orchestrators/platform.engine.ts`
- `packages/engine/src/orchestrators/commerce.engine.ts`
- `packages/types/src/engine/index.ts`
- `supabase/migrations/00001_initial_schema.sql`
- New migration required
- Refund route/tests under `apps/ops-admin` and `apps/web/src/app/api/webhooks/stripe`
Recommended fix: Choose one canonical value, preferably `partially_refunded` for consistency with engine status, update DB constraint, generated types, engine code, UI labels, webhook handling, and tests.
Risk if ignored: Refund failures, accounting drift, customer support escalations, Stripe/local reconciliation mismatch.
Validation steps:
- Add migration and regenerate DB types.
- Add unit tests for full and partial refund.
- Replay partial refund webhook in Stripe sandbox.
- Verify order row, ledger entries, refund case, and Stripe event state.

### 2. Resolve payment capture model mismatch

Issue: Lifecycle and transition side effects use authorization/capture language, including capture at pickup, but checkout PaymentIntent creation does not set `capture_method: 'manual'`.
Impact: Business, finance, and support may misunderstand when funds are captured. If the intended model is manual capture, code does not currently prove it.
Files involved:
- `apps/web/src/app/api/checkout/route.ts`
- `apps/web/src/app/api/webhooks/stripe/route.ts`
- `packages/types/src/engine/transitions.ts`
- `packages/engine/src/orchestrators/order-creation.service.ts`
- `packages/engine/src/orchestrators/master-order-engine.ts`
- `packages/engine/src/services/ledger.service.ts`
Recommended fix: Decide immediate capture at checkout or manual capture at pickup. Then align PaymentIntent parameters, webhook events, transition names, ledger entries, docs, and tests.
Risk if ignored: Incorrect customer charging behavior, failed capture expectations, inconsistent ledger/audit events.
Validation steps:
- Stripe sandbox checkout.
- Inspect PaymentIntent capture status.
- Run pickup/delivery transitions.
- Verify ledger and order history event names match actual Stripe behavior.

### 3. Validate order customer/storefront foreign keys

Issue: `orders_customer_id_fkey` and `orders_storefront_id_fkey` are restored as `NOT VALID` in `00031_security_hardening.sql` and are not later validated.
Impact: Existing orphaned rows can remain undetected; production referential integrity is incomplete.
Files involved:
- `supabase/migrations/00031_security_hardening.sql`
- New migration required
- Any cleanup scripts if orphan rows exist
Recommended fix: Add a pre-validation orphan check and cleanup policy, then add a migration that validates both constraints.
Risk if ignored: Broken order history, dashboard errors, payout/refund failures from missing customer/storefront references.
Validation steps:
- Run orphan detection queries in staging.
- Apply cleanup or block deployment if orphan rows exist.
- Apply `ALTER TABLE orders VALIDATE CONSTRAINT ...`.
- Replay migrations from scratch and upgraded DB.

### 4. Fix smoke E2E port conflict

Issue: `pnpm test:smoke` failed before executing because port 3002 was already in use and Playwright config disallows `reuseExistingServer`.
Impact: No browser smoke validation was completed; frontend runtime readiness is unknown.
Files involved:
- Playwright smoke config files
- Package scripts in `package.json`
- Ops app dev server config
Recommended fix: Make smoke tests use isolated configurable ports, detect/stop only owned test servers, or allow `reuseExistingServer` in local mode while keeping CI deterministic.
Risk if ignored: Runtime regressions can pass typecheck/test/build but fail in browser.
Validation steps:
- Run `pnpm test:smoke` from clean terminal.
- Run again with a preexisting unrelated process on 3002 and verify safe behavior.
- Upload smoke screenshots/traces as CI artifacts.

### 5. Add P0 regression tests for payment/refund/lifecycle

Issue: The current test suite passes, but did not catch the partial refund DB constraint mismatch or capture-model ambiguity.
Impact: Critical payment/lifecycle bugs can survive unit tests.
Files involved:
- `packages/engine/src/orchestrators/*.test.ts`
- `apps/web/src/app/api/webhooks/stripe/__tests__`
- `apps/ops-admin/src/app/api/orders/[id]/refund` tests
- Migration/type tests
Recommended fix: Add focused tests that assert DB-allowed payment statuses match engine-written statuses and PaymentIntent capture mode matches documented lifecycle.
Risk if ignored: Same class of status/finance drift can recur.
Validation steps:
- Introduce failing tests before implementation.
- Verify tests fail on current bug and pass after fix.

Validation steps:

- Existing staging upgrade passes.

### 2. Stripe sandbox contract suite

Issue: Stripe flows were reviewed statically but not executed against sandbox in this review.
Impact: Payment, webhook, refund, payout, and reconciliation behavior remains unproven.

Files involved:

- `packages/engine`

Recommended fix: Add a staging E2E scenario: customer checkout, chef accept/start/ready, dispatch driver, driver pickup/deliver, order complete, payout/ledger visible, customer review.
Risk if ignored: External QA can hit broken cross-app wiring.
Validation steps:

### 4. Supabase RLS role tests

Issue: RLS was reviewed statically only.
Impact: Cross-tenant data leakage or blocked legitimate access can be missed.
Files involved:
- `supabase/migrations`

Recommended fix: Add tests using anon, customer, chef, driver, ops, and service-role clients against staging/test DB.
Risk if ignored: Data exposure or broken app functionality in production.

### 5. Dispatch and driver payout consistency

Issue: Dispatch delivery creation uses a fixed base fee driver payout calculation while ledger computes driver payable from order delivery fee.
Impact: Driver-facing payout may not match finance ledger.
Files involved:

Recommended fix: Make one source of truth for driver payout calculation and reuse it in dispatch, ledger, earnings, and payouts.
Risk if ignored: Driver disputes, accounting mismatch, ops reconciliation noise.
Validation steps:
- Unit tests for delivery fee/tip/surge cases.
- End-to-end completed delivery verifies delivery row, ledger, earnings, and payout run match.

Impact: Wrong scheduler target can skip idempotency/processor-run tracking or use incomplete logic.

## P2 - Before Production

Issue: Missing Upstash config in production-like environments degrades to in-memory rate limiting.
Impact: Multi-instance deployments can be abused by spreading requests across instances.
Files involved:
- `packages/utils/src/rate-limit/index.ts`
- App health/readiness checks
- Deployment env docs
Recommended fix: Add readiness failure or high-severity alert when production rate limiting is degraded.
Risk if ignored: Brute force, checkout abuse, upload abuse, and driver location spam are easier.
Validation steps:
- Staging without Upstash reports not-ready.
- Staging with Upstash reports distributed provider.

### 2. Private storage for sensitive media

Issue: Profile/menu/storefront images and delivery proof images use public URLs; delivery proof can be sensitive.
Impact: Delivery evidence or personal images may be exposed beyond intended users.
Files involved:
- `apps/web/src/app/api/upload/route.ts`
- `apps/chef-admin/src/app/api/upload/route.ts`
- `apps/driver-app/src/app/api/upload/route.ts`

Validation steps:
- Unauthenticated request cannot fetch proof/document media.
- Authorized ops/driver/customer flows can fetch signed URLs only where allowed.

Files involved:
