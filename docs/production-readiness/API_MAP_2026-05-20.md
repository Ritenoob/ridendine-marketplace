# API Map - 2026-05-20

Source: route files under `apps/*/src/app` and handler method search for `export async function GET|POST|PATCH|PUT|DELETE`. Generated `.next` route type files were ignored. State-changing route protection was additionally checked with `pnpm audit:guards`, which reported 120 scanned routes, 13 allowlisted routes, and 0 unguarded state-changing routes.

Status labels describe verified source wiring, not full runtime behavior. Complex external-service routes remain PARTIAL or NEEDS MANUAL TEST until staging execution proves them.

Post-fix update: checkout now explicitly uses automatic Stripe capture, partial refund status mismatch is fixed in engine/migration, the legacy SLA cron route is retired, and driver delivery proof upload now returns signed URLs from a private bucket. Routes still require staging/browser/service validation where marked PARTIAL or NEEDS MANUAL TEST.

## Cross-Cutting API Findings

PASS:
- State-changing route guard audit passed.
- Health routes exist for all apps.
- Stripe webhook routes use raw event handling and signature verification.
- Processor routes have token validation helpers.

PARTIAL:
- Read-route authorization was reviewed by route/source patterns, but not fully penetration-tested.
- Response shapes are not uniformly documented as a contract.
- Some routes use admin client behind app-layer guards; RLS cannot protect those operations.
- Complex routes need staging tests with real Supabase and Stripe.

Remaining runtime gaps:
- Refund routes need Stripe sandbox replay after the partial refund DB/code fix.
- Smoke E2E route coverage needs to be rerun after the isolated-port fix.
