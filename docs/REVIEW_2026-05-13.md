## Phase 1 — Codebase Overview

### Purpose
Ridendine is a **chef-first, end-to-end food-delivery marketplace** where Ridendine itself is the merchant of record. Home chefs onboard via a chef portal, list dishes, accept orders; an internal dispatch engine assigns drivers; customers order through a public marketplace. The active business target (per [docs/plans/2026-05-06-ridendine-platform-review.md](docs/plans/2026-05-06-ridendine-platform-review.md):13) is a **soft launch in Hamilton, ON** — i.e. this is a pre-launch product, not yet in production at scale.

Four distinct user populations:
- **Customers** — browse, cart, checkout, track order ([apps/web](apps/web))
- **Chefs** — storefront/menu/orders/payouts ([apps/chef-admin](apps/chef-admin))
- **Drivers** — offers, deliveries, earnings ([apps/driver-app](apps/driver-app), PWA)
- **Ops staff** — order ops, dispatch, finance, promos, surge, support, team ([apps/ops-admin](apps/ops-admin))

### Architecture shape
**Modular monolith in a monorepo.** No microservices, no message bus, no separate worker tier. There is *one* shared business-logic package — [packages/engine](packages/engine) — that all four Next.js apps import and call into from their API route handlers.

Seams (in order of how cleanly they're enforced):
1. **App ↔ engine.** API routes are thin; orchestrators + services in `@ridendine/engine` hold the domain logic. ([packages/engine/src/index.ts](packages/engine/src/index.ts))
2. **Engine ↔ DB.** Engine talks to Supabase exclusively through `@ridendine/db` repositories. ([packages/db/src/repositories](packages/db/src/repositories))
3. **DB ↔ Supabase.** A generated `database.types.ts` (5,254 LOC) plus hand-merged `database.merged.ts`.
4. **Auth.** `@ridendine/auth` ships a Next middleware factory and server helpers, consumed by each app's `src/middleware.ts`.

The engine package is the **load-bearing center** — every meaningful business operation flows through it (orders, dispatch, kitchen, commerce, support, platform, ops, risk, payouts, loyalty, referral, stripe, ledger, reconciliation, tax, geocoding, surge, ETA).

### 2.2 — Database / SQL (29 findings; the worst ones first)

- **[D1 — Stray SQL files outside the migrations system]** 🤖 — [supabase/fix_permissions.sql](supabase/fix_permissions.sql), [supabase/make_userid_nullable.sql](supabase/make_userid_nullable.sql) — Neither is in `supabase/migrations/`, so `supabase db push` will NOT apply them. The seed script depends on `make_userid_nullable.sql` having run (it inserts chef/driver/customer rows with `NULL user_id`); without it, `pnpm db:seed` will fail with FK constraint violations on a fresh DB. If `db:reset` is run in production, both files are silently skipped. — **C / S** (Fix: rename to `00099_*.sql` and `00100_*.sql`, move into `migrations/`).

- **[D2 — `audit_logs_insert_all` policy allows audit forging]** 🤖 — `supabase/migrations/00010_contract_drift_repair.sql` — Policy `FOR INSERT TO authenticated WITH CHECK (true)` lets any authenticated user insert arbitrary audit_logs entries, including impersonating `actor_role = 'super_admin'`. Audit integrity is broken. — **C / S** (drop the policy; only service-role inserts via triggers/RPC).

- **[D3 — Missing RLS policies on `menu_item_availability`]** 🤖 — Table created in `00001`, RLS enabled in `00003`, but no policies defined → in PostgREST with RLS enabled and no policies, the table is **inaccessible to authenticated and anon**. This is the opposite of what was claimed; it might actually be inaccessible (good) or accidentally open via a service-role-default grant. Verify before acting. — **C / S** (verify behavior and add explicit chef-write policy if needed).

- **[D4 — `orders` has no FK constraint on `customer_id` or `storefront_id`]** 🤖 — `00001_initial_schema.sql` — Both are `UUID NOT NULL` but lack `REFERENCES`. Deleting a customer/storefront leaves orphan orders; FK joins in code can produce inconsistent results. — **C / M** (add FKs with `ON DELETE RESTRICT` in a new migration; verify no orphans exist).

- **[D5 — Audit trigger relies on `current_setting('app.actor_type')`]** 🤖 — `00014_fix_audit_trigger.sql` — `actor_type` defaults to `'system'` when the session-local var isn't set. Almost no app code sets it, so 95%+ of audit rows attribute to `'system'` instead of the real actor. — **H / M** (switch to `auth.uid()` lookup, or add helper that always sets the GUC).

- **[D6 — Promo / driver-presence / notification column duplication via sync triggers]** 🤖 — `00010`, `00016` introduce BEFORE-INSERT/UPDATE sync triggers keeping `valid_from/expires_at` ↔ `starts_at/valid_until` ↔ `max_uses/usage_limit` ↔ `times_used/usage_count` aligned. App code reads/writes inconsistently: [apps/web/src/app/api/promos/validate/route.ts:32](apps/web/src/app/api/promos/validate/route.ts:32) selects `used_count` while [apps/web/src/app/api/checkout/route.ts:49-50](apps/web/src/app/api/checkout/route.ts:49) expects `times_used`. Trigger covers this today, but is brittle. — **H / M** (pick canonical names, update all callers, drop the deprecated columns + triggers next migration).

- **[D7 — Three columns for order lifecycle: `status`, `engine_status`, `public_stage`]** 🤖+🔍 — `master-order-engine.ts` only writes `engine_status` and legacy `status` ([packages/engine/src/orchestrators/master-order-engine.ts:272](packages/engine/src/orchestrators/master-order-engine.ts:272)); `public_stage` is updated by trigger `orders_sync_public_stage_trg` (migration `00019`). So `public_stage` IS kept in sync via trigger. But this means: any code path that bulk-UPDATEs `engine_status` via service-role bypasses the trigger only if the trigger isn't `AFTER UPDATE` — verify. — **H / S** (read the trigger; if `BEFORE UPDATE`, we're fine. If `AFTER`, audit any direct-SQL paths.)

- **[D8 — Inconsistent money types]** 🤖 — `orders.subtotal/total/tip/tax/delivery_fee` are `DECIMAL(10,2)` (dollars); `ledger_entries.amount_cents` is `INTEGER`; `platform_accounts.*_cents` is `BIGINT`; `driver_payouts.amount` is DECIMAL but `stripe_amount_cents` is BIGINT. Two unit systems coexist with manual conversions every time money crosses domains. Rounding-loss bugs are the highest-likelihood class of bugs in this codebase. — **H / L** (post-beta migration: standardize on integer cents everywhere).

- **[D9 — Chef cannot CRUD `menu_item_option_values` under RLS]** 🤖 — Migration `00002` adds only a public SELECT policy. Modifier-tier menus are read-only after creation through the chef portal. — **H / S** (add chef-update policy keyed on storefront ownership).

- **[D10 — No RLS policies on `stripe_events_processed` and `stripe_reconciliation`]** 🤖 — RLS is `ENABLE`d but no policies → table is effectively closed to authenticated/anon, only service-role can read/write. This is the **secure** outcome (good); but undocumented. — **M / S** (add explicit comment + a finance-staff SELECT policy for the ops UI).

- **[D11 — `ledger_entries.order_id` FK should be `ON DELETE SET NULL`]** 🤖 — Migration `00020` correctly drops `NOT NULL` but leaves the FK as default `NO ACTION`. If you ever hard-delete an order, the delete fails. — **M / S**.

- **[D12 — Idempotency tables grow unbounded]** 🤖 — `checkout_idempotency_keys`, `stripe_events_processed`, `stripe_reconciliation` have no TTL. Operational debt, not a beta blocker. — **L / S** (add a 90-day cleanup job).

- **[D13 — No CHECK constraints on money columns]** 🤖 — `orders.subtotal/total/tip/tax/delivery_fee` could legally be negative. App-layer validation should catch it, but defense in depth is cheap. — **L / S** (`CHECK (subtotal >= 0)` etc.).

- **[D14 — `seed.sql` brittle on re-run]** 🤖 — Uses `ON CONFLICT (id) DO NOTHING`; once stale data is in, re-seeding does not refresh it. — **L / S** (TRUNCATE-and-reseed pattern, or use `DO UPDATE`).

### 2.3 — Engine (`@ridendine/engine`) findings

- **[E1 — Stripe Connect Express payouts are not actually triggered]** 🤖+🔍 — [packages/engine/src/services/stripe-webhook-finance.ts](packages/engine/src/services/stripe-webhook-finance.ts) handles `transfer.created`, `payout.paid`, `payout.failed` (the *result* side) — but no code in [payout.service.ts](packages/engine/src/services/payout.service.ts) calls `stripe.transfers.create(...)`. So the **execution side of chef/driver payouts is stubbed**: `executeChefRun()`/`executeDriverRun()` insert `chef_payouts`/`driver_payouts` rows marked `'bank'` rail and ledger entries, but no money actually moves through Stripe. Cross-reference: only `payouts-*-preview` crons are scheduled in [apps/ops-admin/vercel.json](apps/ops-admin/vercel.json) — there is no `payouts-execute` cron. So manual ops triggering is implicit. This may be intentional for beta (Ridendine cuts manual cheques) but it must be confirmed. — **C / L** if launch requires automatic payouts; **Info** if manual is fine for beta.

- **[E2 — Refunds break double-entry ledger]** 🤖+🔍 — [apps/web/src/app/api/checkout/route.ts:649+](apps/web/src/app/api/checkout/route.ts) cancels the order on payment failure but does not call `ledgerService.recordRefund()`. Successful payments DO record the order-payment ledger entries; cancellations after capture leak. — **H / M** (wire `recordRefund` into the cancel path + the ops refund route).

- **[E3 — Driver matching has no failure escalation]** 🤖 — [packages/engine/src/orchestrators/dispatch-orchestrator.ts:64-172](packages/engine/src/orchestrators/dispatch-orchestrator.ts:64) — If `findEligibleDrivers()` returns `[]` (no online drivers, or no presence records), the order stays in `DISPATCH_PENDING` indefinitely with no alert. No retry loop, no system_alerts insert, no notification to ops. — **H / M** (emit `system_alerts` row + ops push notification after N retries fail).

- **[E4 — Driver presence has no TTL]** 🤖 — `driver-matching.service.ts` filters by `driver_presence.status = 'online'` but doesn't check `last_location_at` freshness. A driver who closes the tab stays "online" in the matching pool with stale coords. — **M / S** (add `AND last_location_at > now() - interval '90 seconds'`).

- **[E5 — HST rate read from constants, not DB]** 🤖 — [packages/engine/src/constants.ts:6-11](packages/engine/src/constants.ts) exposes `HST_RATE = 13`. [apps/web/src/app/api/checkout/route.ts:138](apps/web/src/app/api/checkout/route.ts) imports it directly instead of going through `TaxConfigService`. So the ops-admin "tax rates" UI in `platform_settings` is bypassed at checkout. — **M / S** (use `TaxConfigService.getRates()`; failure path → constants).

- **[E6 — Concurrent payout cron races]** 🤖 — Two invocations of `executeChefRun()` from the manual ops button can both progress past `previewChefRun()` to `recordPayout()`. The unique `idempotency_key` on ledger inserts prevents financial loss, but the *result* is `payout_runs` rows with `errors[]` populated — operator confusion. — **M / S** (insert a `payout_runs` row with `status='processing'` + unique index on `status='processing'` for this rail/window).

- **[E7 — Risk engine is rules-only]** 🤖 — `risk.engine.ts` blocks oversized/wrong-currency orders; no velocity or geo-anomaly checks. Acceptable for beta; do not invest now. — **Info**.

- **[E8 — Notification senders fail silently when API key absent]** 🤖 — [packages/engine/src/core/email-provider.ts:87-89](packages/engine/src/core/email-provider.ts:87), [sms-provider.ts](packages/engine/src/core/sms-provider.ts) skip when `RESEND_API_KEY`/`TWILIO_*` unset. DB notifications still written. For beta, customers will only see in-app notifications unless these keys are in Vercel env. — **M / S** (verify Vercel env vars before beta; add a `/api/health` echo that confirms providers are wired).

### 2.5 — `apps/chef-admin` (chef)

- **[A1 — Middleware enforces session only, not chef role]** 🔍 — [apps/chef-admin/src/middleware.ts:3-14](apps/chef-admin/src/middleware.ts:3) creates the shared `createAuthMiddleware` factory which checks for a Supabase session and nothing else. **Any authenticated user (customer, ops, driver) can navigate to chef-admin URLs.** The downstream `getChefActorContext()` does block at the API boundary, so practical exploitation is limited to seeing the chrome of pages, but it's still wrong defense-in-depth. — **H / S** (extend the auth middleware factory to accept `requiredRole` and reject non-chef sessions; same fix applies to ops-admin and driver-app).

- **[A2 — Pending chefs can use the dashboard]** 🤖 — `getChefActorContext` returns success for `status='pending'` per the subagent. Means a chef who signed up but hasn't been approved can hit storefront/menu/orders APIs. Mostly harmless (no real orders exist for an unapproved chef), but should be a clean 403. — **H / S**.

- **[A3 — `chef_payout_accounts` setup not gated on approval]** 🤖 — A pending chef can hit `/api/payouts/setup` and create a Stripe Connect Express account, populating `chef_payout_accounts.stripe_account_id`. Doesn't move money (Connect onboarding doesn't complete payouts), but pollutes Stripe with throwaway accounts. — **M / S**.

- **[A4 — Realtime order channel relies on RLS only]** 🤖 — Channel filter `storefront_id=eq.${storefrontId}` is just a client hint; RLS on `orders` is the real boundary. If RLS misconfig ever happens (it has happened in this codebase 5+ times based on migration names), chefs receive cross-leak realtime events. — **M / S** (extra `if (newOrder.storefront_id !== mine) return` in the handler).

- **[A5 — Audio notification fails silently if asset missing]** 🤖 — [apps/chef-admin/src/components/orders/orders-list.tsx:213-226](apps/chef-admin/src/components/orders/orders-list.tsx) plays `/sounds/new-order.mp3` then `.catch()` swallows. Verify the asset is in `public/sounds/` and audible. — **M / S** (visual cue should not depend on audio working).

- **[A6 — Auto-rejection after 8min timeout is client-side only]** 🤖 — Same file:232-246. If the chef closes the tab, no rejection fires; order sits until ops intervention. — **M / M** (server-side SLA timer should auto-cancel and reassign).

- **[A7 — Kitchen address placeholder accepted on storefront create]** 🤖 — [apps/chef-admin/src/app/api/storefront/route.ts:123-144](apps/chef-admin/src/app/api/storefront/route.ts) writes `address_line1: 'To be updated', postal_code: 'V0V 0V0'` when the chef has no kitchen yet. ### 2.6 — `apps/ops-admin` (internal)
